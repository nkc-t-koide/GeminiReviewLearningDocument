PRを作成する

GitHubで feature/beginner ブランチからPull Requestを作成します。PRを作ると、STEP4で用意したWorkflowが起動し、STEP5のPython処理がAIレビューを実行します。

1

Compare & pull request ボタンを確認する

feature/beginner ブランチをpushすると、GitHubのリポジトリ画面に Compare & pull request ボタンが表示されることがあります。
このボタンからPull Request作成画面へ進めます。

補足： ボタンが表示されない場合は、上部の「Pull requests」タブから新しくPull Requestを作成しても構いません。
2

baseブランチをdevelopにする

Pull Request作成画面では、baseブランチを develop、compareブランチを feature/beginner に設定します。
今回の教材では、main ではなく develop に向けてPull Requestを作成します。

basedevelop
comparefeature/beginner
注意：baseをmainにしない

GitHub Pagesはmainブランチから公開しているため、作業中のコードを直接mainへ入れないようにします。
レビュー用のPull Requestは、feature/beginner から develop に向けて作成します。

3

Create pull request を押す

base が develop、compare が feature/beginner になっていることを確認したら、タイトルを入力して Create pull request を押します。

タイトル例

サンプルシューティングを追加

補足： 説明欄は空でも作成できますが、何を変更したかを簡単に書いておくと確認しやすくなります。
注意：baseを確認してから作成する

base が main のままだと、公開用ブランチへ向けたPull Requestになってしまいます。
この教材では、feature/beginner から develop へPull Requestを作成します。

PR作成後にAIレビューが動きます

Pull Requestを作成すると、GitHub Actionsが起動し、Gemini APIによるAIレビューコメントが投稿されます。

AIレビューコメントを確認する

PRのConversationに、github-actions bot からAIレビューコメントが投稿されていれば成功です。Summary、Critical、Major、Minor などの形式でレビュー結果が表示されます。

AIレビューコメントの例

AIがコード差分と review_rules/review_policy.md をもとに、問題点や改善案をコメントします。

Checksの成功を確認する

PR画面のChecksで、AI PR Review が成功していることを確認します。All checks have passed と表示されていれば、Workflowは正常に完了しています。

確認するポイント
  • AI PR Review のチェックが成功している
  • github-actions bot からコメントが投稿されている
  • コメントの中に Summary / Critical / Major / Minor / Good Points / Questions が表示されている
  • beginner / normal / pro でレビューの内容や深さが変わっている

レビュー結果を比較する

同じゲーム実装でも、レビュー条件を変えるとAIが注目するポイントや説明の深さが変わります。

beginner命名、基本的なバグ、読みやすさなど、初心者が理解しやすい観点を中心に確認します。
normal可読性、責務分離、例外ケース、危険な処理など、実践的な観点を広く確認します。
pro設計、拡張性、依存関係、保守性など、より踏み込んだ観点で確認します。
比較の見方：指摘の数だけでなく、どの問題を重く見ているか、修正案が具体的かも比べてみましょう。

AIレビューコメントが投稿されないとき

Workflowが成功しているのにPRコメントが投稿されない場合は、GitHub Actions の権限設定を確認します。

  1. リポジトリの Settings を開く
  2. Actions → General を開く
  3. Workflow permissions を確認する
  4. 「読み取りおよび書き込み権限」を選択する
  5. Save を押す
補足： この設定が読み取り専用のままだと、GITHUB_TOKEN がPRにコメントを投稿できず、GitHub API status: 403 になることがあります。
APIエラーの見方
  • 403 は権限不足やコメント投稿権限の不足を確認します。
  • 429 はAPIの利用制限に達している可能性があります。少し時間を置いて再実行します。
  • 503 は一時的なサービス側の問題の可能性があります。Workflowを再実行して確認します。
